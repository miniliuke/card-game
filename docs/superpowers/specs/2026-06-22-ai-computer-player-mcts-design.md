# AI 电脑玩家与 MO-ISMCTS 设计

- 日期：2026-06-22
- 状态：设计已在访谈中逐项确认，等待书面规格复核
- 适用项目：Arcana Table（Rust 2024 + Bevy 0.18）

## 1. 背景

当前项目已经具备完整的《璀璨宝石》纯 Rust 规则层和可交互的 Bevy 对战界面：

- `src/rules/` 负责状态、合法性、行动执行、弃牌/贵族续接、计分和终局；
- `src/battle.rs` 持有 `GameState`，将输入转换为规则行动，并消费 `GameEvent` 播放动画；
- 对局固定为 2 人，两个玩家目前都由同一位用户操作；
- `GameState` 可克隆，适合模拟搜索，但完整状态包含牌堆顺序和所有保留牌身份，不能直接交给公平 AI。

本功能把玩家 1 固定为人类、玩家 2 固定为电脑，并实现一个最长异步思考 1 秒的公平隐藏信息 MCTS 玩家。

## 2. 目标与验收标准

### 2.1 目标

1. 玩家 0 是人类且先手，玩家 1 是电脑。
2. 电脑自动完成正常行动、超限弃牌和贵族选择。
3. 每个独立决策最多异步搜索 1 秒，不阻塞 Bevy 主线程。
4. AI 不读取真实牌堆顺序或对手的盲抽保留牌。
5. 搜索核心不依赖 Bevy，可用固定种子和固定迭代数独立测试。
6. 所有 AI 结果仍通过现有规则入口提交，不绕过合法性校验。

### 2.2 可验证的成功标准

- AI 回合期间界面保持响应，且人类输入被锁定。
- AI 在正常行动、弃牌和贵族选择三类上下文中只提交合法决策。
- 搜索超时、取消、任务异常或结果过期均不会卡死对局。
- 固定种子的集成测试可让 AI 完成整场对局，无 panic、非法状态或无法推进的回合。
- 手动强度基准中，交换先后手后，固定迭代预算的 MCTS 对随机玩家胜率不低于 65%。

## 3. 非目标

- 3–4 人或多个电脑玩家；
- 难度选择界面；
- 多核并行 MCTS；
- 跨回合复用搜索树；
- 机器学习、策略网络或自对弈训练流水线；
- 长期玩家行为建模；
- 存档、持久化对局回放或网络对战；
- 修改基础游戏胜负规则。

首版只有一个 `Normal` 配置：单搜索线程、每决策 1 秒、最多 100,000 个树节点、rollout 最多推进 60 个完整回合。

## 4. 方案比较

### 4.1 方案 A：完全信息 UCT MCTS

直接克隆 `GameState`，让搜索读取真实牌堆和全部保留牌。实现最简单，搜索结果也最稳定，但电脑会系统性利用人类无法获得的信息。该方案违反公平性要求，否决。

### 4.2 方案 B：单观察者 ISMCTS

从电脑视角脱敏状态，每轮搜索重新采样未知牌。它能阻止电脑读取人类盲抽牌，但在模拟人类行动时，树统计仍可能间接利用电脑自己的私有牌，让模拟对手表现得像知道电脑手牌。该方案仍有信息泄漏，否决。

### 4.3 方案 C：多观察者、行动者相对的 ISMCTS

采用多观察者信息集 MCTS（MO-ISMCTS），并在每个行动节点按当前行动者可见的信息生成信息集键、合法决策与启发式先验。模拟中对行动者不可见的牌会按其观察重新采样，使任何玩家的决策都不依赖其无权查看的信息。

该方案实现成本最高，但能满足公平隐藏信息、理性对手和可测试信息边界，故采用。

## 5. 总体架构

AI 分为纯 Rust 搜索层和 Bevy 调度层。

```text
完整 GameState（仅主线程持有）
        │
        │ AiObservation::from_game(observer)
        ▼
脱敏 AiObservation + DecisionContext + MctsConfig
        │
        │ Bevy AsyncComputeTaskPool（单个后台任务）
        ▼
MO-ISMCTS ── determinize / legal decisions / UCT / rollout / evaluate
        │
        ▼
AiSearchResult { decision, metrics, request token }
        │
        │ 主线程验证 request token，等待事件与动画清空
        ▼
apply_action / resume（规则层唯一写入口）
        │
        ▼
现有 GameEvent、动画与界面刷新流程
```

建议文件布局：

```text
src/ai/
  mod.rs               公共类型和重导出
  decision.rs          决策上下文、AiDecision、合法决策枚举
  observation.rs       脱敏观察和信息集键
  determinization.rs   从观察构造一致的可能世界
  evaluation.rs        非终局局面评估
  rollout.rs           轻量随机化 rollout 策略
  mcts.rs              MO-ISMCTS 树、搜索循环、取消与指标
src/rules/
  player.rs            ReservedCard / ReserveOrigin
  actions.rs           legal_actions 与保留牌模型适配
  events.rs            保留事件来源信息
src/battle.rs           玩家控制器、后台任务和结果提交集成
```

搜索文件保持小而聚焦。`battle.rs` 已较大，本功能只加入调度胶水；MCTS、观察、采样和评估不得进入该文件。

## 6. 规则层数据模型调整

### 6.1 保留牌来源

现有 `PlayerState::reserved_cards: Vec<CardId>` 无法区分市场保留和牌堆盲抽。改为：

```rust
#[derive(Clone, Copy, PartialEq, Eq, Debug, Hash)]
pub enum ReserveOrigin {
    Market,
    BlindDeck(CardLevel),
}

#[derive(Clone, Copy, PartialEq, Eq, Debug, Hash)]
pub struct ReservedCard {
    pub card_id: CardId,
    pub origin: ReserveOrigin,
}
```

规则语义：

- 从市场保留的卡牌身份属于公共历史；所有玩家和 AI 都可使用；
- 从指定等级牌堆盲抽的卡牌只对持有者可见；其他玩家只知道等级；
- 对手界面中的盲抽牌显示牌背；
- 已购牌仍全部公开；
- `CardReserved` 事件携带 `ReserveOrigin`，但事件不会传给后台 AI，界面按观察者决定是否显示卡面。

### 6.2 合法行动 API

规则层新增：

```rust
pub fn legal_actions(state: &GameState, player: PlayerId) -> Vec<PlayerAction>;
```

它穷举并通过现有 `validate_action` 过滤：

- 可用普通色中的三个不同颜色组合；
- 银行至少有 4 个的同色双拿；
- 所有可保留市场位置和非空牌堆；
- 所有买得起的市场牌和己方保留牌。

AI 不复制正常行动规则。弃牌组合和贵族候选与 `ActionOutcome` 相关，由 `decision.rs` 在模拟包装状态上枚举。

## 7. 决策与模拟状态

AI 必须处理三类决策：

```rust
pub enum DecisionContext {
    MainTurn,
    Discard { excess: u8 },
    ChooseNoble { candidates: Vec<NobleId> },
}

pub enum AiDecision {
    Action(PlayerAction),
    Discard(TokenSet),
    ChooseNoble(NobleId),
}
```

MCTS 使用包装状态：

```rust
pub struct SimulationState {
    pub game: GameState,
    pub context: DecisionContext,
}
```

`SimulationState::apply_decision` 是搜索内部唯一状态转移入口：

- `MainTurn` 调用 `apply_action`；
- `Discard` 调用 `resume(..., Resume::DiscardTokens(...))`；
- `ChooseNoble` 调用 `resume(..., Resume::ChooseNoble(...))`；
- 将返回的 `ActionOutcome` 映射为下一 `DecisionContext`；
- `Complete` 时以规则层已经推进后的 `current_player` 开始下一个完整回合；
- `winner.is_some()` 时成为终局状态。

弃牌枚举生成总量恰好等于 `excess`、且不超过玩家持有量的所有 `TokenSet`。贵族选择只能来自规则层返回的候选集合。

每类决策都获得独立的最长 1 秒搜索预算。若 AI 主行动产生弃牌或贵族选择，后续上下文会新建一次搜索，而不是共享主行动剩余时间。

## 8. 观察与隐藏信息边界

### 8.1 类型边界

后台任务只接收 `AiObservation`，绝不接收完整 `GameState`。这是安全约束而非编码约定。

观察包含：

- 所有公共银行筹码、市场牌、贵族、当前玩家和终局状态；
- 所有玩家公开的筹码、已购牌、贵族和保留牌数量；
- 所有市场来源保留牌；
- 观察者自己的全部保留牌；
- 对手盲抽保留槽位的等级，不含 `CardId`；
- 每个等级牌堆剩余数量；
- 标准卡牌和贵族的静态定义索引。

观察明确不含：

- 实际牌堆顺序；
- 对手盲抽牌的身份；
- 与当前状态无关的玩家行为历史。

### 8.2 Determinization 守恒

每次 MCTS 迭代从观察构造一个可能世界：

1. 从标准 90 张牌中移除市场牌、已购牌、已知保留牌；
2. 按等级拆分剩余候选牌；
3. 为对手每个未知盲抽槽从对应等级无放回采样；
4. 将余牌洗牌并按观察中的剩余数量组成三個牌堆；
5. 复制所有公开筹码、分数来源、贵族和回合字段；
6. 校验每张卡恰好位于市场、牌堆、保留区或已购区之一。

同一搜索的每轮迭代都重新采样，避免某个偶然牌序主导结果。

### 8.3 多观察者节点

树节点使用当前行动者的 `InfoSetKey`，而非完整模拟状态。键包含该行动者可见观察的稳定摘要、决策上下文和行动历史位置，但不含其不可见牌。

在行动者节点：

1. 从该行动者观察重新采样其不可见信息；
2. 只用该观察生成合法决策和启发式排序；
3. 用该行动者的信息集键查找或创建节点；
4. 把选出的合法决策应用到当前迭代世界；
5. 当行动者切换时，为新行动者重复上述过程。

必须用成对测试证明：只改变某玩家不可见的牌，不会改变该玩家的 `InfoSetKey`、合法决策集合或启发式先验。

## 9. MO-ISMCTS 搜索

### 9.1 配置

```rust
pub struct MctsConfig {
    pub time_limit: Duration,          // Normal: 1 s
    pub iteration_limit: Option<u32>,  // 测试使用；正常游戏为 None
    pub max_nodes: usize,              // 100_000
    pub max_rollout_turns: u16,        // 60
    pub exploration: f32,              // sqrt(2)
    pub rollout_epsilon: f32,          // 0.15
}
```

正常游戏以时间、节点和取消标记为边界。确定性测试不使用墙钟，改用固定种子与固定迭代数。

### 9.2 单次迭代

1. 从根观察生成新的 determinization；
2. **选择**：已展开行动使用 UCT；根 AI 节点最大化根胜率，对手节点最小化根胜率；
3. **扩展**：从未访问合法行动中按启发式顺序扩展一个；
4. **Rollout**：双方使用同一套观察安全的轻量策略推进；
5. **结算**：终局返回胜 `1`、负 `0`；达到 60 个完整回合时使用局面评估；
6. **回传**：更新本次路径的访问次数和根玩家总收益。

UCT 利用项在根玩家节点使用 `mean_reward`，在对手节点使用 `1 - mean_reward`。探索项使用：

```text
sqrt(2) * sqrt(ln(parent_visits) / child_visits)
```

未访问行动总是先于 UCT 行动扩展。达到 100,000 节点后不再分配节点，但继续在已有树上选择、rollout 和回传，直到时间截止。

根节点最终选择访问次数最多的合法行动。平均胜率仅用于诊断，不作为最终选择的第一排序键；访问次数相同时按平均收益、再按稳定行动序排序。

### 9.3 启发式 rollout

rollout 不是纯随机：

- 15% 概率从全部合法决策均匀选择，维持策略多样性；
- 85% 概率按轻量行动分数加权抽样；
- 得分买牌、触发贵族、增加有效永久折扣和避免终局落后获得正权重；
- 无价值占满保留区、制造低效筹码和帮助对手立即终局获得负权重；
- 所有合法行动始终保留非零概率。

弃牌优先归还与市场、己方保留牌近期费用关联最弱的筹码。贵族选择优先保留对后续发展颜色最匹配、同时最能阻断对手的贵族。

## 10. 截断局面评估

终局只优化胜率，不用分差替代胜负。仅当 rollout 达到深度上限时使用评估函数。

评估先计算根玩家相对对手的六个归一化优势，均限制在 `[-1, 1]`：

| 特征 | 权重 | 含义 |
|---|---:|---|
| 分数优势 | 0.35 | 当前分差除以 15 |
| 永久折扣 | 0.20 | 五色发展卡总数与覆盖度优势 |
| 贵族进度 | 0.15 | 对当前贵族需求的最佳完成比例优势 |
| 即时购买力 | 0.15 | 当前可买牌的得分、折扣和贵族推进价值优势 |
| 筹码效率 | 0.10 | 持有筹码对可见/保留目标缺口的匹配度，扣除超限风险 |
| 保留牌质量 | 0.05 | 己方保留目标价值与占满低价值保留槽惩罚 |

加权优势 `a` 限制到 `[-1, 1]`，最终估计值为 `0.5 + 0.5 * a`。所有权重位于集中配置结构中，首版不引入学习或自动调参。

## 11. 异步调度与 Bevy 集成

### 11.1 玩家控制器

```rust
pub enum PlayerController {
    Human,
    Computer(MctsConfig),
}
```

首版对局配置固定为 `[Human, Computer(Normal)]`。界面标记玩家 0 为 `YOU`，玩家 1 为 `CPU`，玩家 0 先行动。

### 11.2 请求与过期保护

每个搜索请求携带：

```rust
pub struct AiRequestToken {
    pub match_id: u64,
    pub state_version: u64,
    pub player: PlayerId,
    pub context_kind: DecisionContextKind,
    pub request_id: u64,
}
```

规则状态每次成功 `apply_action` 或 `resume` 后增加 `state_version`。主线程只接受与当前 token 完全匹配的结果；其他结果直接丢弃并在仍需 AI 决策时重新搜索。

### 11.3 生命周期

- 人类完整回合提交且 `current_player` 变为电脑后，立即创建观察和后台任务；上一回合动画可继续播放。
- 若人类仍在弃牌或贵族选择阶段，不启动 AI。
- AI 的每个主行动、弃牌或贵族选择各自创建最长 1 秒的新任务。
- 搜索完成后，结果可暂存，但只有事件队列为空、动画结束且 token 仍有效时才能提交。
- AI 思考或等待提交期间锁定人类输入，状态栏显示 `AI THINKING…`。
- AI 决策通过 `apply_action` 或 `resume` 提交，并复用现有事件、动画、终局和界面刷新流程。
- 返回菜单、重新开局或状态版本变化时设置原子取消标记；搜索循环和 rollout 定期检查该标记与截止时间。

后台使用 Bevy `AsyncComputeTaskPool`，单任务内部不再创建线程。首版不做树内并行，以保持实现简单和固定迭代测试可复现。

## 12. 保底与错误处理

搜索开始前先用启发式策略计算一个合法保底决策。

- 时间到期：返回当前根节点访问次数最多的行动；
- 尚无完整迭代：返回保底决策；
- 达到节点上限：停止扩展，继续利用现有树；
- 任务取消：返回取消状态，不提交行动；
- 任务 panic、采样不一致或内部错误：记录错误，主线程重新验证并提交保底决策；
- 结果过期：静默丢弃并按当前观察重新搜索；
- 规则层拒绝 AI 决策：记录错误、递增状态版本并从新观察重新搜索，禁止无限重复同一失效结果。

所有提交前均再次调用规则层合法性校验。AI 错误不得修改 `GameState`，也不得让对局永久停在思考状态。

## 13. 随机性、复现与诊断

### 13.1 随机种子

正常游戏的 AI 种子由对局种子、决策序号和 AI 玩家编号派生。由于正常游戏按墙钟停止，不保证不同机器重新计算出相同行动。

测试使用固定种子和固定迭代数，结果必须确定。未来对局回放应记录已选决策并重放，不重新执行 MCTS。本功能不实现持久化回放。

### 13.2 指标

正式界面只显示 `AI THINKING…`。调试日志可配置输出：

- 请求 token、随机种子和决策上下文；
- 耗时、迭代数、节点数和取消原因；
- 每个根行动的访问次数与平均收益；
- 最终行动和是否使用保底。

这些指标由 `AiSearchMetrics` 随结果返回，不进入规则状态。

## 14. 测试策略

### 14.1 规则层

- 市场保留和盲抽保留正确记录 `ReserveOrigin`；
- 购买保留牌、计分、事件和上限逻辑在模型迁移后保持不变；
- `legal_actions` 的每个结果都通过 `validate_action`；
- 对典型局面，所有预期合法行动都被枚举且无重复。

### 14.2 观察与 determinization

- 人类观察不包含 CPU 盲抽牌，CPU 观察不包含人类盲抽牌；
- 自己的盲抽牌始终可见，市场来源保留牌对双方可见；
- 改变不可见牌不会改变观察值和 `InfoSetKey`；
- 每次采样保持卡牌唯一、等级正确、槽位数与牌堆数守恒；
- 多个种子能产生不同可能世界，但公开字段保持一致。

### 14.3 搜索算法

- UCT 在根节点最大化、在对手节点最小化根收益；
- 未访问行动先扩展；
- 回传访问数与收益正确；
- 根选择优先访问次数；
- 固定种子和迭代数返回确定结果；
- 节点上限、取消和零完整迭代走正确降级路径；
- 只改变当前行动者不可见信息，不改变其合法行动和先验排序。

### 14.4 集成与性能

- AI 能处理 `MainTurn`、`Discard` 和 `ChooseNoble`；
- 过期 `request_id`、错误玩家、错误状态版本的结果不会提交；
- AI 搜索期间人类输入不进入行动队列；
- 退出 Battle 会取消任务；
- 多个固定种子的电脑对局可以结束且没有非法状态；
- 强度基准交换先后手，对随机玩家胜率至少为 65%；
- 手动运行确认搜索期间动画、窗口和菜单保持响应。

耗时强度与墙钟测试标记为 `#[ignore]` 或独立 benchmark 命令，不拖慢日常 `cargo test`。

## 15. 实施顺序

1. 迁移保留牌模型并保持现有规则测试全绿；
2. 增加规则层合法行动枚举；
3. 建立 AI 决策和模拟状态；
4. 建立脱敏观察与 determinization；
5. 实现评估和 rollout；
6. 实现可取消、可限时的 MO-ISMCTS；
7. 加入 Bevy 后台任务和 CPU 回合调度；
8. 隐藏对手盲抽牌并更新玩家标签；
9. 完成集成、强度和手动响应性验证。

每一步先写失败测试，再写最小实现；每个阶段形成独立提交。

## 16. 已确认的产品决策

- 固定双人 Human vs CPU；
- Human 为玩家 0 且先手；
- 每个决策异步思考最多 1 秒；
- AI 不读取隐藏信息；
- 使用多观察者、行动者相对重采样的 ISMCTS；
- UCT 探索常数默认 `sqrt(2)`；
- 根行动按访问次数选择；
- 启发式 rollout 和截断评估，不用纯随机 rollout；
- 理性零和对手模型；
- 单后台线程、无跨回合树复用；
- 100,000 节点上限和 60 完整回合 rollout 上限；
- 正常游戏按墙钟，测试按固定迭代；
- 无难度菜单、无持久化回放、无机器学习。
